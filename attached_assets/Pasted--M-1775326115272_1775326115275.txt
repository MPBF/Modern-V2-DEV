برومبت استعلام الخصائص المطلوبة لتطبيق الموبايل
مقدمة
أنا الوكيل المسؤول عن تطبيق الموبايل لنظام MPBF. نحتاج تنفيذ عدة خصائص جديدة في التطبيق. أرجو الإجابة على كل الأسئلة التالية بالتفصيل حتى أتمكن من بناء الواجهات بشكل متكامل مع السيرفر.

1. نظام الحضور والانصراف (4 حالات)
التطبيق حالياً يدعم حالتين فقط: حضور وانصراف. نحتاج إضافة استراحة وعودة من الاستراحة.

أسئلة:
هل يوجد endpoints للاستراحة والعودة؟ مثلاً:

POST /api/attendance/break-start — بدء استراحة
POST /api/attendance/break-end — العودة من الاستراحة
إذا لا، هل ستضيفها؟ وما هو شكل الطلب والاستجابة المتوقع؟

هل حقل status في سجل الحضور يدعم حالات الاستراحة؟ حالياً التطبيق يرسل status: 'حاضر'. هل يوجد حالات مثل:

'حاضر' — في العمل
'استراحة' — في استراحة
'غائب' — غائب
هل سجل الحضور (AttendanceRecord) يحتوي على حقول الاستراحة؟ مثل:

break_start: string | null — وقت بدء الاستراحة
break_end: string | null — وقت العودة من الاستراحة
break_duration_minutes: number | null — مدة الاستراحة
أم أن الاستراحات تُسجل في جدول منفصل؟

هل يمكن أخذ أكثر من استراحة في اليوم؟ إذا نعم، كيف يتم تخزينها (مصفوفة من الاستراحات مرتبطة بسجل الحضور؟)

ما هو التسلسل المتوقع؟

حضور → استراحة → عودة → انصراف
أم
حضور → (استراحة → عودة) × عدة مرات → انصراف

هل الموقع GPS مطلوب لتسجيل الاستراحة والعودة أيضاً؟

2. الطلبات — الخصائص الناقصة
أ) تحويل الطلب إلى الإنتاج
حالياً أوامر الإنتاج تُنشأ تلقائياً مع الطلب. لكن نحتاج زر "تحويل للإنتاج" منفصل.

هل يوجد endpoint لتحويل طلب موجود إلى الإنتاج؟ مثل:

POST /api/orders/{id}/convert-to-production
أم يتم فقط تغيير حالة الطلب إلى in_production عبر PATCH /api/orders/{id}/status؟
عند تحويل طلب للإنتاج، هل يتم إنشاء أوامر الإنتاج تلقائياً من السيرفر أم يجب أن يرسلها التطبيق؟

هل يمكن تحويل طلب بحالة waiting فقط أم أي حالة؟

ب) حذف الطلب
DELETE /api/orders/{id} — هل هذا الـ endpoint يعمل؟ وما هي الشروط؟
هل يمكن حذف طلب في حالة in_production؟
هل يتم حذف أوامر الإنتاج المرتبطة تلقائياً؟
هل هو حذف فعلي (hard delete) أم تغيير حالة إلى cancelled؟
ج) طباعة الطلب
هل يوجد endpoint لتوليد PDF أو تقرير قابل للطباعة للطلب؟ مثل:

GET /api/orders/{id}/print — يرجع PDF
GET /api/orders/{id}/export — يرجع بيانات مهيأة للطباعة
إذا لا، هل يمكنك إضافته؟ ما هي المعلومات التي يجب أن يتضمنها (بيانات العميل، المنتجات، الكميات، تواريخ التسليم)؟

هل النظام يدعم مشاركة الطلب عبر WhatsApp أو البريد الإلكتروني؟

3. المستودع — عمليات الإدخال والإخراج
حالياً المستودع في التطبيق للعرض فقط. نحتاج إضافة إنشاء سندات.

أسئلة عامة:
هل يوجد endpoints لإنشاء سندات المستودع؟ مثل:

POST /api/warehouse/vouchers — إنشاء سند جديد
ما هو شكل الطلب المتوقع؟ مثال تقريبي:

{
  "type": "rm_in",
  "items": [
    { "item_id": "...", "quantity": 100, "unit": "kg", "notes": "..." }
  ],
  "supplier_id": 5,
  "notes": "..."
}

ما هي الحقول المطلوبة لكل نوع سند؟

سند إدخال خام (rm_in): هل يحتاج supplier_id؟ purchase_order_number؟
سند إخراج خام (rm_out): هل يحتاج ربط بأمر إنتاج (production_order_id)؟
سند إدخال تامة (fp_in): هل يتم ربطه بأمر إنتاج تلقائياً؟
سند إخراج تامة (fp_out): هل يحتاج ربط بطلب عميل (order_id)؟ customer_id؟
ما هي العناصر التي يمكن إدخالها في كل نوع سند؟

rm_in/rm_out: من أين أحصل على قائمة المواد الخام؟ هل هي /api/inventory أم /api/items أم endpoint آخر؟
fp_in/fp_out: هل العناصر هي منتجات العملاء (/api/customer-products) أم شيء آخر؟
هل يوجد نظام موافقات على السندات؟ (مثلاً: السند يحتاج موافقة مدير المستودع)

هل يمكن تعديل أو إلغاء سند بعد إنشائه؟

4. الإعدادات والتعريفات — إضافة/تعديل/حذف
حالياً الشاشة للعرض فقط. نحتاج CRUD كامل.

أسئلة لكل كيان:
العملاء (Customers)
هل توجد endpoints للـ CRUD؟
POST /api/customers — إنشاء عميل. ما هي الحقول المطلوبة والاختيارية؟
PUT /api/customers/{id} — تعديل عميل
DELETE /api/customers/{id} — حذف عميل. هل يمكن حذف عميل لديه طلبات؟
منتجات العملاء (Customer Products)
هل توجد endpoints للـ CRUD؟
POST /api/customer-products — إنشاء منتج. ما هي الحقول المطلوبة؟
PUT /api/customer-products/{id} — تعديل
DELETE /api/customer-products/{id} — حذف
ما هي الحقول المطلوبة عند الإنشاء؟ (customer_id, size_caption, width, thickness, raw_material, master_batch_id, is_printed, punching, cutting_unit, unit_weight_kg?)
المستخدمين (Users)
هل توجد endpoints لإدارة المستخدمين من الموبايل؟
POST /api/users — إنشاء مستخدم
PUT /api/users/{id} — تعديل
DELETE /api/users/{id} — حذف/تعطيل
ما هي الحقول المطلوبة؟ (username, password, display_name, display_name_ar, role_id, section_id, phone, email?)
الأدوار (Roles)
هل توجد endpoints للأدوار؟
GET /api/roles — عرض الأدوار
POST /api/roles — إنشاء دور
PUT /api/roles/{id} — تعديل دور وصلاحياته
ما هي قائمة الصلاحيات الكاملة المتاحة في النظام؟
الأقسام (Sections)
CRUD endpoints:
POST /api/sections — إنشاء. الحقول: name, name_ar, description?
PUT /api/sections/{id} — تعديل
DELETE /api/sections/{id} — حذف
الأصناف/العناصر (Items)
CRUD endpoints:
POST /api/items — إنشاء. الحقول: name, name_ar, category_id, unit?
PUT /api/items/{id} — تعديل
DELETE /api/items/{id} — حذف
التصنيفات (Categories)
CRUD endpoints:
POST /api/categories — إنشاء
PUT /api/categories/{id} — تعديل
DELETE /api/categories/{id} — حذف
الموردين (Suppliers)
CRUD endpoints:
POST /api/suppliers — إنشاء. الحقول: name, name_ar, phone, email, city?
PUT /api/suppliers/{id} — تعديل
DELETE /api/suppliers/{id} — حذف
الماكينات (Machines)
CRUD endpoints:
POST /api/machines — إنشاء
PUT /api/machines/{id} — تعديل (تغيير الحالة، السعة)
DELETE /api/machines/{id} — حذف
5. التقارير — التحسينات المطلوبة
تقرير الحضور والانصراف:

هل يوجد endpoint يرجع ملخص الحضور لفترة معينة؟ مثل:
GET /api/reports/attendance?from=...&to=...
عدد أيام الحضور، الغياب، التأخير، إجمالي ساعات العمل، متوسط ساعات العمل
تقرير الطلبات المفصل:

هل /api/reports/orders يرجع بيانات كافية؟ ما هو شكل الاستجابة بالتفصيل؟
هل يدعم فلترة بالتاريخ أو العميل أو الحالة؟
تقرير المستودع:

هل يوجد تقرير حركة المخزون لفترة معينة؟
هل /api/warehouse/reports/summary يكفي أم يحتاج تحسين؟
تصدير التقارير:

هل يمكن تصدير أي تقرير كـ PDF أو Excel؟
هل يوجد endpoints للتصدير مثل GET /api/reports/orders/export?format=pdf؟
6. أسئلة عامة
الصلاحيات — ما هي القائمة الكاملة؟ أحتاج معرفة كل الصلاحيات لتقييد الوصول حسب الدور:

من يمكنه إنشاء طلبات؟
من يمكنه حذف طلبات؟
من يمكنه إنشاء سندات مستودع؟
من يمكنه إدارة المستخدمين والتعريفات؟
من يمكنه رؤية التقارير؟
هل يوجد نظام تنبيهات/إشعارات push notifications جاهز؟ هل يرسل السيرفر إشعارات عند:

إنشاء طلب جديد
اكتمال أمر إنتاج
مخزون منخفض
طلب صيانة جديد
هل يوجد validation rules على السيرفر؟ مثلاً:

الحد الأدنى/الأقصى للكمية في أمر الإنتاج
التحقق من تكرار اسم العميل
التحقق من صحة رقم الهاتف
المطلوب
أرجو الإجابة على كل سؤال برقمه، مع توضيح:

هل الـ endpoint موجود أم يحتاج إنشاء؟
شكل الطلب (Request Body) المتوقع
شكل الاستجابة (Response) المتوقع
أي قيود أو شروط
هذا سيمكنني من بناء جميع الواجهات بشكل متكامل مع السيرفر دون الحاجة لتعديلات لاحقة.
